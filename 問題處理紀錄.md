### 問題 #001 - Sitemap.xml 無法被 Google Search Console 擷取
**日期**：2025-09-02  
**網站**：https://joechiboo.github.io

#### 🔍 問題描述
- Google Search Console 無法正確擷取 sitemap.xml
- 擔心 sitemap 格式或存取有問題

#### 🛠 診斷過程
1. **檢查 sitemap 可存取性**
   ```bash
   curl -I https://joechiboo.github.io/sitemap.xml
   # 結果：HTTP 200 OK - 正常可存取
   ```

2. **驗證 XML 格式**
   - 檢查本地檔案：`public/sitemap.xml`
   - XML 結構正確，符合標準格式

3. **分析 URL 結構**
   - 發現使用 Hash 路由：`https://joechiboo.github.io/#/about`
   - 這是主要問題來源

#### ✅ 根本原因
**Hash 路由影響 SEO**：
- 搜尋引擎對 `#/path` 格式的支援較差
- Google 可能無法正確索引 hash fragment 後的內容
- Sitemap 中的 hash 路由可能被搜尋引擎忽略

#### 🔧 處理方案

**立即解決**：
1. 修改 sitemap.xml，移除 `#` 符號：
   ```xml
   <!-- 改為 -->
   <loc>https://joechiboo.github.io/about</loc>
   <!-- 而非 -->
   <loc>https://joechiboo.github.io/#/about</loc>
   ```

2. 在 Google Search Console 重新提交 sitemap

3. 等待 1-7 天讓 Google 重新處理

**長期優化**：
- 考慮改用 History Mode 路由而非 Hash Mode
- 實作 prerendering 或 SSG 提升 SEO 效果

#### 📊 處理結果
- [x] 創建 SEO-Sitemap-Guide.md 文件
- [x] 分析問題根本原因
- [x] 實際修改 sitemap.xml（已完成，移除所有 # 符號）
- [ ] 重新提交 Google Search Console（待執行）
- [ ] 追蹤後續索引狀況（待觀察）

**修改內容**：
- 移除所有 URL 中的 `#` 符號
- 更新 lastmod 為當前日期 (2025-09-02)
- 保持其他設定不變

#### 💡 經驗學習
1. **Sitemap 可存取 ≠ SEO 有效**：檔案能下載不代表搜尋引擎能正確處理內容
2. **Hash 路由的 SEO 限制**：SPA 應用需特別注意路由模式對 SEO 的影響
3. **診斷順序很重要**：先確認基本可存取性，再檢查內容格式，最後分析結構問題

### 問題 #002 - 專案資料夾雙層巢狀導致 npm install 失敗
**日期**：2026-06-07
**網站**：https://joechiboo.github.io

#### 🔍 問題描述
- 在 `C:\Personal\Project\joechiboo.github.io` 執行 `npm install` 報錯 `ENOENT: ... package.json`
- 路徑看起來「怪怪的」，同名資料夾出現兩層

#### 🛠 診斷過程
1. 列出目錄結構，發現雙層巢狀：
   ```
   joechiboo.github.io\            ← 外層（非 git repo，只有 .claude/）
   └── joechiboo.github.io\        ← 內層才是真正的專案 + git repo
       ├── .git\  package.json  src\ ...
   ```
2. 確認 `npm install` 是在外層執行 → 該層沒有 `package.json`，且失敗時順手在外層留下一個 98 bytes 的空 `package-lock.json`
3. 搬移前確認內層 git 狀態：`git status` 乾淨、`main...origin/main` 同步、改動皆已 commit/push

#### ✅ 根本原因
- 專案被多包了一層資料夾（外層只是空殼），npm 指令在錯誤的層級執行

#### 🔧 處理方案
1. 刪除外層誤產生的空 `package-lock.json`
2. 將內層所有內容（含隱藏檔與 `.git`）上移一層，消除巢狀：
   ```powershell
   # 先把 shell 工作目錄移出內層，避免 Windows 鎖住資料夾
   Get-ChildItem -Force -Path $inner | Move-Item -Destination $outer -Force
   Remove-Item $inner -Force
   ```
3. `.claude\` 衝突處理：保留 repo 追蹤的內層版本（`settings.local.json`），並保留外層 session 的 `settings.json`

#### 📊 處理結果
- [x] 巢狀結構已消除，`.git` 歷史與遠端同步皆完好
- [x] 搜尋/類別 filter 功能已 commit + push（`03603b0`）
- [ ] **待驗證**：在新單層路徑執行 `npm install && npm run dev`，確認搜尋框與類別 filter 實際渲染正常
- [ ] **待決定**：未追蹤的 `.claude/settings.json` 要 `git add` 納管，還是加入 `.gitignore` 忽略
- [ ] **待留意**：HEAD 的 `2b58646`（ParkFee）commit 把搜尋功能一起帶上去、訊息卻只寫 ParkFee，疑似有另一個 session 也在動此 repo，後續操作前先確認沒有並行作業

#### 💡 經驗學習
1. **npm ENOENT 多半是層級問題**：先確認當前目錄是否真的有 `package.json`，而非急著重裝
2. **搬移 git repo 前先確保工作目錄乾淨且已推送**：此時搬移風險最低，`.git` 不在意絕對路徑
3. **Windows 搬移資料夾要先讓所有 shell 離開該目錄**，否則會被鎖住無法移除
4. **同名巢狀資料夾易誤導**：clone 或建立專案時注意不要多包一層
